Eliot's Weekly MongoDB World Challenge #1:

The Aggregation Rock Star Challenge


Eliot's Weekly Challenge is a competition for developers in the run up to MongoDB World. Test out your skills and win prizes every week. Find out more.

 

MongoDB makes it easy to quickly explore data and get meaningful results. In this challenge, you're going to be exploring a data set of all known meteorite landings which has been compiled by The Meteoritical Society.

 

Since 1976, scientists have trekked through the inhospitable Transantarctic Mountains in search of fallen meteorites. As part of the Antarctic Search for Meteorites program (ANSMET), their mission is to learn about our solar system through meteorites which have fallen to Earth. These intrepid scientists are real rock stars! Let's join them by taking a look at what they found.

Challenge


1. Sign up for a free MongoDB Atlas account.
2. Create a free M0 MongoDB cluster.
3. Import the meteorite landing data set (meteorites.json) which was compiled by The Meteoritical Society.
4. Create a 2dsphere geospatial index on the location field (you will need this).
5. Use MongoDB Compass to build a single aggregation pipeline to return the following data:

 

For each decade from 1970 until now, find the average mass and standard deviation of all meteorites within a 1000 km radius of the McMurdo Antarctic research station (lon: 166.6863, lat: -77.8419).

 

Completed? Submit your answer below.

The Data Set


The data set provided in meteorites.json contains information about over 45,000 meteorites that have fallen to Earth. Each document has the following fields:


• name: The name given to the meteorite.
• recclass: The class of the meteorite. Meteorites can be classified into a wide range of categories based on the type of materials of which they are composed. You don't need to know anything about meteorite classification to complete the challenge.
• mass: The mass of the meteorite in grams.
• fall: Either "Found" or "Fell". A meteorite is marked as "Fell" if it was observed falling from space. Otherwise it is marked as "Found".
• year: The year in which the meteorite fell or was found. This is a timestamp in a string. For example, the year 2000 is represented as "2000-01-01T00:00:00".
• location: A geoJSON point showing the location the meteorite was found.

We require the output of your aggregation to be in this specific format. You can use this example result to test your aggregation pipeline is correct.


Note that due to floating-point inaccuracies, the last couple of decimal places in your standard deviation may be different from this example. Don't worry if your numbers are slightly off. We will consider answers correct if they are within 10 significant digits. We will also be inspecting the aggregation pipelines you submit to ensure correctness.

2. Import the meteorite landing dataset

First you will need to download the data set. You can download the file here. If you're on a Unix system you can download the file using the following command:

 

wget https://raw.githubusercontent.com/mongodb-developer/weekly-challenge-1/master/meteorites.json

 

Using mongoimport
The mongoimport tool is one of the tools installed with MongoDB. You can download MongoDB from the MongoDB Download Center.

 

The following mongoimport command will import the data set into the landings collection of the meteorites database. It will drop the collection if it already exists. You need to replace the URI with your MongoDB Atlas connection string.


mongoimport --uri "mongodb+srv://<username>:<password>@<cluster-id>.mongodb.net/meteorites" -c landings --type json --file ./meteorites.json --drop

3. Create a geospatial index
Using the mongo shell

To create a geospatial index on the location field, first use the meteorites database:


use meteorites


Then run the following command to create the index:


db.landings.createIndex({location: "2dsphere"})


Note that this will create the index in the foreground. You may want to create the index in the background if you are concerned about resource contention in your cluster. If you are using a fresh M0 cluster, creating the index in the background would be unnecessary.

Tips


• The $geoNear stage by default limits the number of documents returned to 100. In order to include all meteorites around McMurdo Station, make sure to set the limit to 25,000 or above.
• The $geoNear stage needs to be the first stage of the pipeline.
• The $geoNear stage requires a 2dsphere index on the location field.
• Some aggregation stages you may find useful:

     • $geoNear
     • $project
     • $bucket
     • $group

• Some aggregation operators you may find useful:

     • $stdDevPop
     • $avg
     • $push
     • $dateFromString

     • $year
